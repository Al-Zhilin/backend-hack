# Transit & Tour Master: Backend Middleware

**Transit & Tour Master** — это высокопроизводительный асинхронный сервер на базе FastAPI, выступающий центральным узлом (Middleware) между пользовательским интерфейсом, удаленной нейросетевой моделью и картографическими сервисами.

Проект решает сложную задачу автоматизации планирования путешествий: от текстового общения с AI до построения математически оптимизированного маршрута с учетом расписаний транспорта.

---

## 🏗 Архитектура системы

Проект работает как "прослойка" (BFF), агрегирующая данные из нескольких источников:
1.  **Frontend:** Отправляет запросы на чат и получает готовые маршруты.
2.  **ML-Server:** Обрабатывает контекст диалога и возвращает структурированные теги через вебхук.
3.  **2GIS API:** Используется для поиска локаций (Catalog API) и построения дорожных графов (Routing API).
4.  **Yandex API:** Используется для точного геокодирования и получения расписаний пригородного/городского транспорта (Raspisanie API).
5.  **MongoDB Atlas:** Хранит временные состояния поиска и финальные сгенерированные туры.

---

## 🛠 Технологический стек

* **Ядро:** FastAPI (Python 3.12+).
* **БД:** MongoDB + Motor (асинхронный драйвер).
* **Сетевой уровень:** HTTPX (асинхронные запросы к API и стриминг).
* **API Интеграции:** 2GIS (Catalog, Routing, Transport), Yandex (Geocoder, Raspisanie).

## 📡 Детальное описание API эндпоинтов

### 1. Взаимодействие с AI (Streaming)

#### `POST /api/chat`
Основной канал связи с нейромоделью.
* **Тип ответа:** `StreamingResponse` (text/plain).
* **Механика:** Сервер не ждет полного ответа от ML-модели, а транслирует чанки текста сразу на фронтенд.
* **Request Body:** JSON, формат которого ожидает удаленный сервер (обычно `{"message": "..."}`).

---

### 2. Сборка тура и Вебхуки

#### `POST /api/receive_tour_data`
Принимает структурированные данные от ML-сервера после анализа предпочтений пользователя.
* **Входные данные:** `TourCollectedData` (login, status, collected_tags).
* **Логика:**
    1.  Получение списка тегов (музеи, парки, кафе).
    2.  Определение типа передвижения (машина/пешеход).
    3.  Запуск фонового процесса обработки маршрута.

#### `GET /api/get_tour/{login}`
Эндпоинт для "опроса" (polling) фронтендом готовности тура.
* **Возвращает:** Объект тура с массивом `route`, временем пути и данными о транспорте.

---

### 3. Инструменты картографии (2ГИС)

#### `POST /api/v1/search-places`
Ищет отель по названию и находит интересные места в заданном радиусе.
* **Params:** `hotel_query` (название), `category` (что искать рядом), `radius` (в метрах).
* **DB Impact:** Очищает коллекцию `temp_places` и записывает туда новые найденные точки для последующего выбора.

#### `POST /api/v1/build-tour`
Финальная сборка маршрута на основе выбранных пользователем ID мест.
* **Механика:**
    1.  Извлекает места из БД по `ObjectId`.
    2.  **TSP Optimization:** Сортирует точки алгоритмом "Ближайшего соседа" для минимизации пути.
    3.  **Routing:** Запрашивает у 2GIS геометрию пути (Selection) для отрисовки линии на карте.

---

### 4. Дополнительные сервисы

* **`GET /api/get-panorama`**: Проксирует запрос к ML-серверу для получения метаданных панорам по координатам.
* **`GET /panorama-image/{file_id}`**: Стриминг бинарных данных изображения панорамы.
* **`GET /api/routes`**: Метод для получения маршрутов между произвольными точками с использованием логики Яндекса и 2ГИС.

## 🧭 Алгоритм работы маршрутизатора (Internal Logic)

1.  **Геокодирование:** Если на вход поступил адрес, `GISService.get_yandex_coords` переводит его в `[lon, lat]`.
2.  **Оптимизация (TSP):** Метод `solve_tsp` принимает стартовую точку и список локаций, возвращая отсортированный список, где каждая следующая точка — ближайшая к предыдущей.
3.  **Транспортная логика:**
    * **Для машин:** Вызывается `get_road_route` с параметром `transport: "car"`. Возвращается время с учетом пробок (jam).
    * **Для пешеходов:** Вызывается `fetch_2gis_transport` для поиска маршрутов на общественном транспорте в радиусе 1500м. Дополнительно может быть задействовано Яндекс.Расписание для межгорода.

## 🚀 Инструкция по развертыванию

### Локальный запуск
1. Установите зависимости: `pip install -r requirements.txt`.
2. Настройте `.env`.
3. Запустите: `python server.py` или `uvicorn server:app --reload`.

### Docker (Production)
```bash
docker build -t transit-master .
docker run -p 8000:8000 --env-file .env transit-master
```

## 💡 Советы для Frontend
* **Стриминг:** При работе с `/api/chat` используйте `Fetch API` и читайте `body.getReader()`, чтобы обновлять UI по мере прихода текста.
* **Поллинг:** После отправки вебхука делайте запрос на `/api/get_tour/{login}` раз в 3 секунды, пока не получите статус `success`.
* **Карта:** Поле `geometry` в ответе маршрута содержит закодированную линию (Polyline). Используйте SDK 2ГИС или Leaflet для её отображения.
