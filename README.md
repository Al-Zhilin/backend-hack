# Transit & Tour Master — Backend Service

Данный сервис представляет собой интеллектуальную прослойку (middleware), которая связывает Frontend-часть приложения, удаленную нейронную модель для анализа предпочтений и географические сервисы (2GIS, Яндекс).

## 🚀 Основной функционал

1.  **AI Proxy Streaming**: Перенаправление запросов к удаленному серверу с нейромоделью в режиме реального времени (Streaming Response).
2.  **Интеллектуальный поиск (2GIS)**: Получение списка туристических локаций на основе тегов, сформированных нейросетью.
3.  **Оптимизация маршрута (TSP)**: Решение "задачи коммивояжера" для сортировки выбранных мест в максимально эффективном порядке посещения.
4.  **Транспортная логистика (Яндекс)**:
    *   Автоматическое определение необходимости общественного транспорта.
    *   Интеграция с Яндекс.Расписаниями для получения актуальных данных о станциях и маршрутах для пешеходов.
5.  **Геокодирование**: Преобразование адресов в координаты и поиск ближайших объектов.
6.  **Хранение данных**: Интеграция с MongoDB Atlas для сохранения черновиков и готовых туров.

## 🔧 Решенные проблемы и исправления (Post-Mortem)

В ходе разработки и деплоя были устранены критические ошибки:

1.  **Fix: ImportError (`_QUERY_OPTIONS`)**:
    *   *Проблема*: Конфликт версий драйверов `motor` и `pymongo` приводил к падению сервера при запуске.
    *   *Решение*: Версии библиотек были строго зафиксированы в `requirements.txt` (`motor==3.3.2`, `pymongo==4.6.3`), что обеспечило их совместимость.
2.  **Стабилизация Runtime**:
    *   *Проблема*: Использование экспериментальной версии Python 3.14 на Render вызывало нестабильность библиотек.
    *   *Решение*: Проект переведен на стабильную версию **Python 3.12.2** через конфигурационный файл `runtime.txt`.
3.  **Оптимизация сборки**:
    *   Внедрена очистка кэша сборки (Clear Build Cache) для предотвращения накопления устаревших зависимостей в виртуальном окружении.

## 🗺 Логика работы API

### 1. Чат с нейромоделью (`/api/chat`)
Поток данных идет напрямую от удаленного сервера к пользователю. Это позволяет видеть ответ модели по мере его генерации, не дожидаясь полного формирования текста.

### 2. Генерация маршрута
После того как нейросеть выделила теги (например: `музеи`, `парки`, `кофейни`), вызывается цепочка:
1.  **2GIS Catalog API**: Поиск объектов в радиусе вокруг отеля по данным тегам.
2.  **TSP Algorithm**: Встроенный алгоритм сортирует найденные точки так, чтобы минимизировать путь.
3.  **Условие транспорта**:
    *   Если выбран тип **"Машина"**: строится автомобильный маршрут через 2GIS Routing.
    *   Если выбран тип **"Пешеход"**: через Яндекс.Расписания запрашиваются ближайшие остановки и расписание транспорта для каждой точки маршрута.

Ниже представлено подробное описание форматов данных и примеры реализации на стороне фронтенда.

### 1. Работа с AI-чатом (Streaming)
**Эндпоинт:** `POST /api/chat`

Это самый специфичный этап. Поскольку сервер проксирует ответ от нейросети по частям, фронтенд не должен ждать завершения запроса. Нужно использовать `ReadableStream`.

**Формат запроса:**
```json
{
  "login": "user123",
  "message": "Я хочу посетить музеи и парки в Москве, передвигаюсь пешком"
}
```

**Пример реализации на JS (Fetch API):**
```javascript
const response = await fetch('https://your-api.render.com/api/chat', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ login: "user", message: "Хочу тур по кафе" })
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const chunk = decoder.decode(value, { stream: true });
    
    // Обновляем интерфейс: добавляем кусочек текста в окно чата
    console.log("Получен кусок текста:", chunk);
    document.getElementById('chat-box').innerHTML += chunk;
}
```

### 2. Поиск мест и создание черновика (`search-places`)
**Эндпоинт:** `POST /api/v1/search-places`

Этот метод используется, когда из сообщения пользователя извлечены "теги" или категории (например, "отели" и "парки").

**Параметры (Query Params):**
* `hotel_query`: Название отеля (точка старта).
* `category`: Категория поиска (тег от нейросети).
* `radius`: Радиус поиска в метрах (по умолчанию 1500).

**Формат ответа:**
Фронтенд получает объект отеля и список "кандидатов" с их внутренними ID из временной базы данных.
```json
{
  "hotel": { "name": "Radisson", "point": {"lat": 55.75, "lon": 37.61} },
  "candidates": [
    { "_id": "65f...", "name": "Парк Горького", "address": "ул. Крымский Вал", "lat": 55.72, "lon": 37.60 },
    { "_id": "65g...", "name": "Музей Гараж", "address": "...", "lat": 55.73, "lon": 37.61 }
  ]
}
```

---

### 3. Построение финального маршрута (`build-tour`)
**Эндпоинт:** `POST /api/v1/build-tour`

Когда пользователь на фронтенде галочками выбрал интересные ему места из списка `candidates`, фронтенд отправляет их ID на сервер для решения **Задачи Коммивояжера (TSP)**.

**Формат запроса (Body):**
```json
{
  "hotel_data": { "name": "Radisson", "point": {"lat": 55.75, "lon": 37.61} },
  "selected_ids": ["65f...", "65g..."]
}
```

**Что делает фронтенд с ответом:**
Сервер возвращает `route` — массив точек, **уже отсортированный** в оптимальном порядке посещения, и `metrics` (время и расстояние).

**Формат ответа:**
```json
{
  "_id": "tour_123",
  "user_hotel": "Radisson",
  "route": [
    { "name": "Radisson", "lat": 55.75, "lon": 37.61 }, 
    { "name": "Музей Гараж", "lat": 55.73, "lon": 37.61 },
    { "name": "Парк Горького", "lat": 55.72, "lon": 37.60 }
  ],
  "metrics": {
    "time_min": 45.5,
    "dist_m": 3400,
    "geometry": "encoded_polyline_string" 
  }
}
```
*Примечание: `geometry` используется для отрисовки линии пути на карте (например, через Leaflet или Mapbox).*

---

### 4. Интеграция с расписанием (Пешеходный режим)
Если в логике приложения определено, что пользователь — **пешеход**, фронтенд может запрашивать данные о транспорте.

**Логика взаимодействия:**
1. Фронтенд вызывает `/api/v1/generate-tour-from-tags` (или расширенный `build-tour`).
2. В ответе для каждой точки маршрута будет поле `yandex_schedule`.
3. **Фронтенд должен отобразить:** "Ближайшее метро/остановка: [Имя]", "Ближайшие рейсы: [Список]".

---

### 5. Обработка ошибок
Сервер возвращает стандартные HTTP-статусы:
*   **404 Not Found**: Отель не найден или мест по тегам нет.
*   **422 Unprocessable Entity**: Ошибка валидации данных (например, переданы неверные координаты).
*   **500 Internal Server Error**: Проблемы с ключами API или внешними сервисами.

**Рекомендация для фронтенда:**
Всегда оборачивайте запросы в `try-catch` и выводите пользователю понятные уведомления (Toast), так как внешние API (2GIS/Yandex) могут быть временно недоступны.
