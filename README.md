# 🗺️ Transit & Tour Master API

Сервер-оркестратор для планирования городских путешествий. Он связывает нейросетевой ИИ, картографические сервисы (2ГИС, Яндекс), сервер компьютерного зрения (OpenCV) для создания панорам и облачную базу данных MongoDB Atlas.

## 🏗️ Архитектура системы

Проект реализован как **асинхронный хаб**, где данные передаются между компонентами по принципу "состояние в базе":
1. **Frontend ↔️ ИИ**: Живой диалог для уточнения предпочтений.
2. **ИИ ➡️ Backend**: Передача "сырых" тегов для подбора мест.
3. **Backend ↔️ Frontend**: Предоставление вариантов на выбор и выдача финального маршрута.
4. **Frontend ↔️ Backend ↔️ OpenCV Server**: Пересылка набора фотографий с устройства пользователя для генерации бесшовных панорам и их последующая отдача.

## 🔄 Логика работы (Workflow)

Процесс разбит на **4 ключевых этапа**:

### Этап 1: Сбор тегов и поиск вариантов (Suggestions)
1. Пользователь общается с ИИ через роут `/api/chat` **Streaming** методом.
2. Когда ИИ собрал данные, он **сам** отправляет JSON на роут `/api/receive_tour_data`.
3. Сервер в фоне ищет реальные места через **2GIS** и сохраняет их в коллекцию `suggestions` в MongoDB.

### Этап 2: Выбор пользователя
1. Фронтенд запрашивает найденные варианты через `/api/get_suggestions/{login}`.
2. Пользователь видит карточки мест и выбирает те, которые ему нравятся.
3. Фронтенд отправляет выбранные объекты на `/api/generate_route`.

### Этап 3: Финализация и логистика
1. Сервер получает выбранные места, решает **задачу коммивояжера** (кратчайший путь) и запрашивает расписания транспорта в **Яндексе**.
2. Результат сохраняется в коллекцию `generated_tours`.
3. Фронтенд забирает готовый план через `/api/get_user_route/{login}` и рисует его на карте.

### Этап 4: Интерактив и создание панорам
1. Находясь на точке маршрута, пользователь может сделать от 1 до 30 фотографий.
2. Фронтенд отправляет фото (Multipart/Form-Data) на `/api/panorama/upload`. Сервер проксирует их на удаленный OpenCV-сервер.
3. OpenCV-сервер сшивает фото, сохраняет результат в Atlas GridFS и возвращает `file_id`. Наш сервер сохраняет привязку этого ID к конкретному месту (`place_id`).
4. При повторном запросе маршрута (`/api/get_user_route/{login}`), в узлах маршрута автоматически появится готовая ссылка `panorama_url`, по которой фронтенд сможет отобразить сшитую панораму.

## 🛠️ Описание Роутов

### 🤖 Для Нейросети
*   **`POST /api/chat`**: Прокси-чат. Фронтенд отправляет текст, получает поток (stream) от ИИ.
*   **`POST /api/receive_tour_data`**: Прием структурированных тегов от ИИ. Запускает фоновый поиск мест.

### 📱 Для Frontend-разработчика (Маршруты)
*   **`GET /api/get_suggestions/{login}`**: Получение списка найденных карточек мест.
    *   *Если 404:* Места еще ищутся (покажите спиннер).
    *   *Если 200:* Отрисуйте массив `suggested_places`.
*   **`POST /api/generate_route`**: Отправка массива выбранных пользователем мест. Сервер ответит "OK" и начнет строить маршрут в фоне.
*   **`GET /api/get_user_route/{login}`**: Получение финального маршрута.
    *   *Результат:* Отсортированные шаги (`route_steps`), расписание транспорта (`schedule`) и, если были сгенерированы панорамы, поле `panorama_url` внутри объектов `from` и `to`.

### 📸 Для Frontend-разработчика (Панорамы)
*   **`POST /api/panorama/upload`**: Отправка исходников панорамы. Принимает `login`, `place_id`, `lat`, `lon` и массив `files` (до 30 штук). Возвращает сгенерированный `file_id`.
*   **`GET /api/panorama/image/{file_id}`**: Прямая отдача (stream) готовой панорамы в формате JPEG (проксируется с OpenCV сервера). Идеально для тегов `<img src="...">`.
*   **`GET /api/panorama/search?lat=...&lon=...`**: Проверка наличия уже существующей панорамы по координатам (поиск готовых панорам других пользователей).

## 🛠 Технологический стек

*   **FastAPI**: Высокопроизводительный асинхронный фреймворк.
*   **Motor (MongoDB)**: Асинхронное взаимодействие с облачной БД Atlas.
*   **HTTPX**: Клиент для асинхронных запросов к внешним API и проксирования файлов.
*   **GIS Services**: 
    *   *Yandex Geocoder*: Преобразование адресов в координаты.
    *   *2GIS Catalog*: Поиск реальных заведений.
    *   *2GIS Routing*: Оптимизация порядка посещения точек.
    *   *Yandex Расписания*: Время работы и маршруты транспорта.
*   **OpenCV (Remote)**: Интеграция с удаленным микросервисом для тяжелых вычислений (сшивка фото `cv2.Stitcher`).

## 🚀 Быстрый старт (Разработка)

### 1. Переменные окружения (`.env`)
Создайте файл `.env` в корне:
```env
YANDEX_GEOCODER_API_KEY=...
YANDEX_RASP_API_KEY=...
DGis_API_KEY=...
MONGO_URI=mongodb+srv://...
DB_NAME=TourProject
BASE_OF_LOCAL_URL=http://localhost:5000
OPENCV_SERVER_URL=http://<IP_ИЛИ_ДОМЕН_OPENCV_СЕРВЕРА>:8000
```

### 2. Запуск
```bash
pip install -r requirements.txt
python server.py
```
Документация будет доступна по адресу: `http://localhost:8000/docs`

## ⚠️ Важные примечания
*   **Polling (Опрос):** Поскольку генерация маршрута, поиск мест и сшивка панорам занимают время, фронтенду необходимо опрашивать GET-роуты с интервалом 2-3 секунды, пока не придет статус 200.
*   **CORS:** На сервере разрешены все источники (`allow_origins=["*"]`), что позволяет фронтенду подключаться без дополнительных настроек.
*   **Проксирование файлов:** Основной сервер не сохраняет картинки на свой жесткий диск, файлы транслируются напрямую в оперативную память и пересылаются на OpenCV сервер для экономии места.
